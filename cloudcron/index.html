CloudCron - Painless Cron Jobs in the Cloud

At polyglot.network, we work on a lot of open-source tools and add features/bugfixes that our customers demand. There are some really nifty command-line tools out there which our customers want to run on a periodic basis. Here are some examples:

  • LightHouse by Google can generate a detailed performance report of any webpage. This is the same tool that Chrome's has built-in as "Performance" tab of its Dev Console. Usage: lighthouse https://yoursite.example.com/
  • Apache Benchmark can load-test your webapp by firing concurrent requests. Usage: usage
  • curl, when run periodically, can act as a health-check.

In almost all of these cases, users want to receive the output of these commands to their email address or to a webhook endpoint, where this could be consumed programmatically. However, setting these up as periodic tasks is too cumbersome for many.

  • You need to provision a server
  • Need to figure out a strategy for maintaining and rotating credentials
  • Install all the dependencies (such as an OS, libraries) and keep them up-to-date
  • Write custom code to send the output to an endpoint such as email or webhook

All this is not hard for developers, but it IS cumbersome and annoying. It should be dead-easy to run batch jobs in the cloud. Since polyglot.network is all about improving FOSS tools, we wanted to provide an easy way to do so. Enter CloudCron, which is built on top of Google's Kubernetes Engine and can run arbitrary docker containers at a specified frequency. Here's all you need to do:

  • Sign up using your email address. We use a magic-link for both registration and login, so there are no passwords to remember or rotate.
  • Tell us the name of the docker image, the name of the command, and the arguments. As an example:
    • Docker image name: google/lighthouse
    • Command name: lighthouse
    • Arguments: https://yourwebsite.example.com/
  • Tell us the job frequency using the crontab syntax:
  • Specify the maximum number of times the job should be run. This puts an upper-bound on the cost.
  • And finally, fill in the endpoint where you'd like to receive the results of each execution. The endpoint can be either an email or webhook.
  • We have advanced options to configure the MIME/type and/or attachments etc.
  • The cost depends on CPU and memory limits that you configure for your job.

This infrastructure is built on top of Google Cloud Components such as Cloud Scheduler, Cloud pub/sub, Cloud Functions, GKE, and CloudSQL to ensure reliability and scalability.

Our customers have been using this for lighthouse, ab, and curl but since we support arbitrary docker images, ANY tool that's available as an image on Docker Hub can be run. Bring your own ideas!
